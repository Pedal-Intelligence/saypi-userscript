# VAD-quality benchmark (#420 item 3)

A repeatable harness that measures the **gating quality** of SayPi's client-side voice
activity detection ‚Äî the foundation #420 calls for before tuning thresholds. It answers,
per preset: how often does the VAD **clip real speech** (false-reject) and how often does
**non-speech open a segment** (false-accept), and at what onset/tail latency ‚Äî for both the
raw VAD and the **#420 admission gate** (`src/vad/segmentAdmission.ts`).

This is the analogue of the server-side ASR WER/RTF benchmark (`saypi-benchmarks`), but it
lives **here** on purpose: the thing being measured is the *client's* gating ‚Äî the exact
Silero v5 model, `@ricky0123/vad-web` frame processor, `VAD_CONFIGS` presets, and SayPi
admission gate that ship in the extension. The harness imports those real modules and runs
the real ONNX model offline, so the numbers reflect the code users actually run ‚Äî not a
reimplementation that could drift.

## Run it

```bash
npm run bench:vad          # replay the corpus through every preset, print the table
```

No microphone, no browser, no network ‚Äî it loads the bundled `silero_vad_v5.onnx` via
`onnxruntime-web` in Node and replays the committed corpus. Deterministic: the model and
corpus are fixed, so the false-reject / false-accept / latency numbers are reproducible.
(Run via `node --experimental-strip-types` so it can import the real TypeScript gate.)

## What it does

For each corpus clip √ó each preset (`highSensitivity`, `balanced`, `conservative`, `none`):

1. Resample + frame the audio with vad-web's `Resampler` (512-sample / 32 ms frames).
2. Run each frame through the **real Silero v5** model ‚Üí per-frame speech probability.
3. Drive vad-web's **real `FrameProcessor`** with the preset's thresholds ‚Üí speech
   start/end segments, exactly as the offscreen/onscreen VAD clients do.
4. Feed each frame's probability into SayPi's **real `SegmentStatsTracker`** and run
   **`admitSegment`** at speech-end ‚Äî the #420 gate ‚Äî tagging each segment admitted/dropped.

Then it aggregates (`lib/metrics.mjs`, unit-tested in `test/bench/vad-metrics.spec.ts`):

- **FRR** (false-reject rate) ‚Äî fraction of *speech* clips that produced no segment.
- **FAR** (false-accept rate) ‚Äî fraction of *non-speech* clips that opened a segment.
- **onset latency** ‚Äî segment start vs. the clip's labelled speech onset.
- **tail latency** ‚Äî segment end vs. labelled speech end (the redemption tail).

‚Ä¶reported twice: **raw** (every segment the VAD opened) and **gated** (only segments the
#420 gate admitted), so the gate's effect on the trade-off is visible.

## The corpus

`corpus/` + `manifest.json` ‚Äî a small **synthetic seed**, regenerated by
`npm run bench:vad:corpus` (needs macOS `say`; output WAVs are committed so *running* the
benchmark needs no tools):

- **Positives (speech):** `say`-voiced confirmations (`yes`/`no`/`okay`/`stop`/`thank you`
  + a sentence) at normal volume, plus deliberately **quiet** variants ‚Äî the soft-speech
  false-reject case #420 worries about ‚Äî at 16 kHz mono PCM16, each with trailing silence.
- **Negatives (non-speech):** deterministically synthesised digital silence, white noise
  (two levels), pink-ish hiss, a 440 Hz tone, and a 60 Hz hum.

### ‚öÝÔ∏è Seed-corpus caveat

These numbers are **directional, not authoritative.** Synthetic `say` speech is clean and
close-mic'd; generated noise is not real ambient; there is **no music or real-world
background**. A trustworthy false-accept/false-reject *knee* needs a labelled corpus of
real recordings (soft/short real speech, caf√©/keyboard/music ambient). Treat this as the
methodology + harness + a first measurement to **expand**, not the final word. Item #420(4)
(host‚Üípreset remapping) should not be decided on synthetic data alone.

One model caveat: the harness runs **every** preset on the Silero **v5** model (what the
shipped `highSensitivity`/`balanced` presets pin). The client's rarely-hit `none` *fallback*
actually uses vad-web's **legacy** model (it sets no `model` field), so the `none` row here
reads as "v5 with no SayPi overrides" ‚Äî useful as a vanilla-v5 baseline, not as a literal
reproduction of that fallback path.

## First findings (seed corpus, 8 speech / 6 non-speech)

| preset | FRR | FAR | onset | tail |
|---|---|---|---|---|
| highSensitivity | 0% | 17% | 60 ms | 425 ms |
| balanced | 0% | 0% | 88 ms | 357 ms |
| conservative | 0% | 0% | 112 ms | 245 ms |
| none (v5 default) | 13% | 0% | 91 ms | 589 ms |

(raw and gated are identical on this corpus ‚Äî see below.)

- **Real speech peaks high ‚Äî even quiet, even on `highSensitivity`.** Every speech clip,
  including the gain-0.18 quiet variants, peaked **0.89‚Äì0.999**. That's a large margin over
  the #420 gate floor (preset threshold + 0.05 = 0.40‚Äì0.45), empirically supporting the
  claim that the gate is safe against clipping *clear* speech. (Caveat: `say` speech is
  clean; far-mic/noisy quiet speech could peak lower ‚Äî exactly what a real corpus would test.)
- **`highSensitivity` is the only preset that false-accepts** (the pink-ish hiss, which
  opened a segment peaking **0.653**), consistent with #420's concern that the most
  trigger-happy preset is the riskiest. `balanced`/`conservative` reject it via their
  stricter threshold + `minSpeechFrames`.
- **The #420 gate neither helped nor hurt on this corpus** (raw FAR == gated FAR, 0 added
  FRR). The gate's drop band is *just above* each preset's opening threshold
  `[threshold, threshold+0.05)`; the seed's negatives are either clearly non-speech (peak ‚âà
  0, never open) or ‚Äî for the pink hiss ‚Äî a **confident** trigger (0.653) the conservative
  gate rightly leaves alone. Demonstrating FAR *reduction* needs borderline real-world
  non-speech that lands in that band. **This is a finding, not a harness bug** (the gate
  logic itself is covered by `test/vad/segmentAdmission.spec.ts`).
- **`none` (Silero v5 defaults) clips short speech** ‚Äî it dropped `stop` because the v5
  default `minSpeechFrames: 9` (288 ms) exceeds that plosive's voiced length. This is
  precisely why SayPi's presets lower `minSpeechFrames` (#420 item 2).

## Files

- `run.ts` ‚Äî CLI entry; orchestrates corpus √ó presets, prints the table, writes
  `report.json` (git-ignored).
- `lib/runner.ts` ‚Äî the offline runner (real v5 model + frame processor + admission gate).
- `lib/metrics.mjs` ‚Äî pure FRR/FAR/latency aggregation (unit-tested).
- `lib/wav.mjs` ‚Äî minimal WAV decode/encode (no deps).
- `generate-corpus.mjs` ‚Äî regenerate the seed corpus.
- `corpus/` ‚Äî committed seed WAVs + `manifest.json` (labels + speech regions).
